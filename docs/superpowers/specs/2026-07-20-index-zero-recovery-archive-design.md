# INDEX-0 异常档案交互设计规格

日期：2026-07-20  
状态：待确认  
项目：Game Design Notes / Vite + React + TypeScript + GSAP + Three.js

## 1. 目标

在现有公开主页中加入一条可被细心访客发现、可通过键盘和触屏完整完成的原创叙事彩蛋：

1. 五段受到污染的文字散落在公开页面不同模块。
2. 访客恢复全部片段后，获得一句完整记录：
   `被删除的记录仍在继续编写`
3. 页脚解锁恢复入口。
4. 输入正确口令后播放一次档案恢复动画。
5. 进入 `CASE-00 / THE UNWRITTEN RECORD` 怪诞剧情页面。
6. 返回主页时保留解谜完成状态，并出现克制的后续变化。

整体身份为“内部异常档案 / 数据删除 / 限制访问 / 损坏网页”。不使用魔法、符文、法术、神秘学粒子、绿色 Matrix 代码雨、SCP 名称、SCP 标志、对象等级或其完整视觉识别。

## 2. 非目标

- 不创建真实后台、账号系统或安全认证。
- 不存放密码、API Key、隐私资料或未公开内容。
- 不收集 IP、姓名、邮箱、设备指纹或其他个人身份信息。
- 不把彩蛋进度上传到 Worker、GitHub 或第三方服务。
- 不永久替换、遮挡或损坏主页正常内容。
- 不以持续抖动、持续乱码或高频闪烁妨碍阅读。
- 不引入新的重型依赖。

## 3. 核心叙事

`INDEX-0` 是一套用于整理项目描述、版本记录和设计笔记的自动归档程序。

某次更新后，它开始生成不存在的项目、没有提交记录的文案，以及发生在创建时间之前的删除请求。系统随后被移除，但它留下的文字仍在公开页面中继续出现。

最后一条完整记录被拆为五段：

1. `被删除`
2. `的记录`
3. `仍在`
4. `继续`
5. `编写`

完整口令：

`被删除的记录仍在继续编写`

## 4. 路由设计

继续使用 GitHub Pages 可用的 hash 路由，不增加服务端重写要求。

### 4.1 路由

- `#/admin`：保留现有伪后台。
- `#/recovery`：异常记录恢复入口。
- `#/case-00`：解锁后的剧情档案。
- `#<work-id>`：保留现有项目档案深链。
- 无 hash：公开主页。

### 4.2 优先级

路由判断顺序固定为：

1. `#/admin`
2. `#/recovery`
3. `#/case-00`
4. 已知项目 ID
5. 公开主页

### 4.3 访问约束

- 未恢复五段线索时直接打开 `#/recovery`：允许进入恢复入口，但界面显示当前完整度，仍可尝试口令。
- 未完成口令验证时直接打开 `#/case-00`：使用 `history.replaceState` 或等价 hash 替换跳转到 `#/recovery`。
- 已完成口令验证时直接打开 `#/case-00`：正常进入剧情页面。
- 现有 `#/admin`、项目 hash、返回键和前进键行为不得被破坏。

## 5. 客户端状态与隐私

### 5.1 存储键

使用单一版本化键：

`gdn:index0:puzzle:v1`

值结构：

```ts
type IndexZeroPuzzleState = {
  version: 1;
  recoveredFragmentIds: string[];
  phraseVerified: boolean;
};
```

允许的片段 ID：

```ts
type FragmentId =
  | "fragment-01"
  | "fragment-02"
  | "fragment-03"
  | "fragment-04"
  | "fragment-05";
```

### 5.2 存储边界

- 只保存五个片段 ID 和口令是否验证。
- 不保存访客输入的口令。
- 不保存时间戳、访问路径、鼠标轨迹、键盘内容或身份信息。
- 写入前校验 ID 白名单并去重。
- `localStorage` 不可用或数据损坏时，回退为当前标签页内存状态；页面仍可完成谜题。
- 未知字段、未知片段 ID 和不匹配的版本号均被忽略或重置。

### 5.3 与访问统计的关系

- 现有匿名访问计数保持原逻辑。
- 解谜进度不发送给统计 Worker。
- `#/recovery` 和 `#/case-00` 视为站内页面，不增加额外的身份或事件追踪。

## 6. 五段公开页面线索

每个线索使用同一个 `CorruptedFragment` 组件。默认是当前文案中的一小段异常文字，不替换整个段落。

| ID | 正确片段 | 初始污染文字 | 放置位置 |
| --- | --- | --- | --- |
| 01 | 被删除 | 被删▮ | Hero 的一行补充说明 |
| 02 | 的记录 | 的记彔 | “近期设计命题”中的一个卡片状态行 |
| 03 | 仍在 | 仍_在 | “设计档案”轮播右侧状态文字 |
| 04 | 继续 | 继｜续 | “近期原型迭代”的版本记录条 |
| 05 | 编写 | 编冩 | “设计笔记”末尾或页脚前的低透明度批注 |

### 6.1 恢复触发

桌面端：

- 鼠标悬停并停留约 700ms：恢复。
- 点击：立即恢复。
- 键盘聚焦后按 Enter 或 Space：立即恢复。
- 键盘聚焦停留约 700ms：恢复。

触屏：

- 单次点击：立即恢复。

每个片段只计入一次，重复操作不会重复增加完整度。

### 6.2 恢复反馈

恢复过程：

1. 污染文字在 220–360ms 内随机替换 2–4 个字符。
2. 字符稳定为正确片段。
3. 邻近位置短暂显示：
   `FRAGMENT 01 / RECOVERED`
4. 标签约 1.6 秒后降低透明度，但正确片段保持可读。

`prefers-reduced-motion: reduce` 下：

- 不播放字符轮换和位移动画。
- 激活后立即显示正确片段和恢复标签。

### 6.3 无障碍

- 片段使用原生 `button` 或等价可操作元素，不使用只有 `div` 的点击行为。
- 未恢复时提供中文 `aria-label`，例如“发现异常片段 01”。
- 恢复后 `aria-label` 更新为“片段 01 已恢复：被删除”。
- 焦点样式清晰，暗金描边与背景对比满足可见性。
- 状态更新通过低打扰 `aria-live="polite"` 区域播报。

## 7. 完整度节点

页面右下角显示低调的全局状态节点：

```text
ARCHIVE INTEGRITY
0 / 5
```

行为：

- 每恢复一段，数字更新一次，并出现短促扫描反馈。
- 桌面端固定在右下安全区，不遮挡导航、按钮和浏览器滚动条。
- 移动端缩小为单行胶囊，位于底部安全区上方；不会覆盖 Contact 操作。
- 节点本身可聚焦，聚焦时显示已恢复的片段编号，不泄露未发现答案。

全部恢复后切换为：

```text
5 FRAGMENTS RECOVERED
UNREGISTERED ENTRY FOUND
```

随后 Contact/页脚中的黑色涂抹在 700–900ms 内从两侧收缩，露出：

`[OPEN RECOVERY INTERFACE]`

入口链接指向 `#/recovery`。

## 8. 恢复入口

### 8.1 页面文案

英文标题：

`RECOVERY INTERFACE`

中文标题：

`异常记录恢复入口`

说明：

`请输入由污染片段组成的完整记录。`

输入框标签：

`RECOVERY PHRASE`

按钮：

`VERIFY RECORD`

### 8.2 输入处理

- 输入前后空白被移除。
- 内部普通空格被移除，允许访客按片段分隔输入。
- 不自动在页面源码中显示完整答案。
- 答案作为纯前端谜题常量存在，因此明确不具备安全认证意义。
- 不把答案发送到任何网络接口。

### 8.3 错误反馈

错误时从下列文案中随机显示一条：

- `记录顺序不正确。`
- `当前文本无法形成有效索引。`
- `缺少至少一个被删除的片段。`
- `该句从未存在于公开版本中。`
- `ERROR ██-05：系统不承认当前输入者。`

交互约束：

- 错误提示使用 `role="alert"`。
- 不清空输入框，便于修正。
- 不播放大幅抖动，只做一次 2–4px 的短位移或边框暗红脉冲。
- 连续错误不会加速闪烁或叠加动画。

### 8.4 正确反馈

显示：

```text
口令匹配。

警告：
该句并未保存于任何已知版本中。
```

随后启动 Archive Recovery Sequence。动画完成后：

1. 将 `phraseVerified` 写为 `true`。
2. hash 替换为 `#/case-00`。
3. 显示剧情档案。

## 9. Archive Recovery Sequence

总时长目标：1.8–2.4 秒。

### 9.1 动画时间线

1. `0.00–0.22s`：画面转黑，中央出现一条细白扫描线。
2. `0.18–0.72s`：扫描线从上到下移动，档案页细线轮廓随扫描显现。
3. `0.32–0.90s`：依次显示：
   - `VERIFYING FRAGMENTS...`
   - `RECONSTRUCTING DELETED SENTENCE...`
   - `SEARCHING VERSION HISTORY...`
4. `0.72–1.12s`：进度条推进至 76% 并停顿。
5. `1.05–1.36s`：暗红文字显示：
   `ERROR: ORIGINAL DOCUMENT DOES NOT EXIST`
6. `1.30–1.72s`：依次显示：
   - `USING OBSERVER MEMORY AS SOURCE`
   - `RECORD RESTORED`
7. `1.62–1.98s`：黑色涂抹从两端覆盖屏幕。
8. `1.90–2.28s`：涂抹向两侧退开，剧情页面淡入。

### 9.2 实现约束

- 使用现有 GSAP 基础设施和 `useGSAP` 清理机制。
- 动画只改变 `transform`、`opacity` 和 CSS 自定义属性，避免持续布局抖动。
- 动画覆盖层带 `aria-live` 状态，完成前锁定重复提交。
- 页面卸载或 hash 改变时正确清理时间线。

### 9.3 减少动态

`prefers-reduced-motion: reduce` 下：

- 显示简洁黑色覆盖层。
- 顺序显示“VERIFYING FRAGMENTS...”和“RECORD RESTORED”。
- 总时长约 300–450ms。
- 不执行扫描、暂停、涂抹位移和页面重新分页。

## 10. CASE-00 隐藏页面

### 10.1 页面头

```text
CASE-00 / THE UNWRITTEN RECORD
第零号档案 / 未被编写的记录
```

状态栏：

```text
ARCHIVE STATE：PARTIAL
DOCUMENT INTEGRITY：87%
OBSERVER STATUS：ACTIVE
SOURCE AUTHOR：████████
REVISION COUNT：UNKNOWN
```

### 10.2 正文

```text
INDEX-0 是一套用于整理项目描述、更新记录和设计文档的自动索引程序。

在版本 ██.██ 中，系统开始生成不存在的修改记录。最初，这些内容被判断为缓存错误。随后，三份档案中同时出现了一段没有作者、没有提交时间、也没有原始文件的文字。

该段文字为：

“被删除的记录仍在继续编写。”

INDEX-0 随后被移除。删除完成后的第十七秒，公开页面出现了第一个污染片段。
```

后续正文：

```text
调查人员认为，INDEX-0 并没有在编写内容。

它只是在寻找能够发现这些内容的人。

每一次污染文字被读取，它都会获得一次新的确认记录。每一次口令被正确输入，CASE-00 的完整度都会提高。

当前完整度：████████ 87%

剩余部分将在本次访问结束后，由当前观察者补全。
```

结尾：

```text
AUTHOR：████████
```

鼠标 hover 或键盘聚焦后短暂显示：

```text
AUTHOR：CURRENT OBSERVER
```

最后以打字动画显示：

```text
你没有进入后台。

你只是完成了它缺少的那一段。
```

### 10.3 页面操作

- `RETURN TO PUBLIC INDEX`：返回主页。
- `CLEAR RECOVERY MEMORY`：可选的低调按钮，清除本地谜题状态并返回主页；操作前要求一次确认。
- 不提供伪造的编辑、删除或后台管理功能。

## 11. 文字样式系统

以下类在 `.index-zero-surface` 范围内定义，避免污染主页已有样式：

- `.archive-heading`：档案主标题；窄体大写、较大字距、轻微重复印刷偏移。
- `.archive-body`：正文；高可读性、适中行宽、较大行距。
- `.system-label`：等宽字体、全大写；状态、编号和机器输出。
- `.redacted`：黑色涂抹；支持静态遮盖、两端收缩和 hover 恢复。
- `.corrupted`：字符错位、替换、重影；只在短暂恢复过程中激活。
- `.margin-note`：文档边缘人工批注；轻微倾斜，移动端回归正文流。
- `.unauthorized-note`：被系统覆盖的批注；出现后由黑条单次覆盖。
- `.ghost-text`：低透明度重复文本，偏移 1–2px。
- `.recovered-text`：恢复完成后的稳定文字与暗金标记。
- `.warning-stamp`：暗红色印章；`RESTRICTED`、`PARTIAL RECORD`、`UNVERIFIED`。

所有装饰性重复文字设置 `aria-hidden="true"`，避免屏幕阅读器重复朗读。

## 12. 文字与文档动效

允许的单次效果：

1. 文档进入视口时扫描线经过一次。
2. 标题首次出现时产生一次轻微重复印刷和 2–4px 归位。
3. 污染文字切换 2–4 个错误字符后恢复。
4. 黑色涂抹从两端收缩。
5. 边缘批注以打字方式出现。
6. 部分批注出现后被系统自动覆盖一次。
7. 状态短暂变化后恢复：
   - `STABLE → UNKNOWN`
   - `PARTIAL → WATCHING`
   - `OBSERVER → SUBJECT`
8. 文档在两个章节切换时产生一次 4–8px 的重新分页位移。

禁止：

- 无限循环的屏幕抖动。
- 高频白闪或暗红闪烁。
- 长时间保留不可读乱码。
- 通过颜色单独表达解锁状态。
- 动画期间改变用户滚动位置。

## 13. 解锁后主页变化

`phraseVerified === true` 时：

1. 完整度节点保持 `5 / 5`。
2. 一个现有项目状态在本次主页首次进入时短暂显示：
   `OBSERVATION COMPLETE`
   约 2.2 秒后恢复原状态。
3. 页脚增加低透明度文字：
   `当前页面已记住本次阅读。`
4. `[OPEN RECOVERY INTERFACE]` 保持可访问。
5. 所有正常项目内容、下载、外链和导航保持原样。

该状态仅来自本地布尔值，不意味着识别了访问者。

## 14. 组件与文件边界

建议新增：

```text
src/
  components/
    index-zero/
      ArchiveIntegrityNode.tsx
      CorruptedFragment.tsx
      RecoveryInterface.tsx
      RecoverySequence.tsx
      CaseZeroArchive.tsx
  data/
    indexZeroArchive.ts
  hooks/
    useIndexZeroPuzzle.ts
  styles/
    index-zero.css
```

建议职责：

- `indexZeroArchive.ts`：片段、错误文案、剧情文本和状态标签的只读数据。
- `useIndexZeroPuzzle.ts`：解析、校验、持久化、恢复片段、验证完成状态。
- `CorruptedFragment.tsx`：单个线索的输入方式、恢复动画和无障碍状态。
- `ArchiveIntegrityNode.tsx`：全局完整度和入口解锁提示。
- `RecoveryInterface.tsx`：口令表单和错误/正确反馈。
- `RecoverySequence.tsx`：独立可清理的 GSAP 时间线。
- `CaseZeroArchive.tsx`：剧情页面布局和一次性文字效果。
- `index-zero.css`：只负责 INDEX-0 视觉层，避免继续膨胀全局样式。

现有 `App.tsx` 只负责路由分流和把片段状态注入公共页面，不承载谜题细节。

## 15. 图像与 Three.js 策略

本功能以文字、涂抹、扫描线和文档排版为主，不需要生成伪装成真实档案或游戏证据的图片。

- 主页保留现有 Three.js 背景。
- 恢复入口和 CASE-00 使用 CSS/GSAP 构建档案轮廓和扫描效果。
- 如视觉验证后确实缺少材质，只允许补充一张原创、无文字、无标志的纸张噪点或打印纹理；不得生成剧情“证据截图”。
- 不加入新的 3D 大场景，避免抢夺文字叙事和移动端性能。

## 16. 响应式规则

桌面端：

- CASE-00 使用主文档列 + 边缘批注列。
- 状态栏可横向排列。
- 完整度节点固定右下。

移动端：

- 文档改为单列。
- 边缘批注进入正文流，不超出视口。
- 状态栏使用两列或单列紧凑网格。
- 完整度节点缩为单行，不遮挡底部操作。
- 所有五段线索无需 hover 即可通过点击恢复。
- 输入框、按钮和入口最小可点击高度为 44px。

### 17.1 构建检查

### 17.2 功能检查

1. 首次访问显示 `0/5`。
2. 五个线索分别可通过鼠标、键盘和触屏恢复。
3. 同一片段重复操作不会重复计数。

## 18. 完成标准

1. 五段线索自然嵌入现有主页，正常阅读不受影响。
2. 细心访客能发现污染，键盘和移动端访客也能完成。
3. 恢复入口、动画和 CASE-00 形成完整叙事闭环。
4. 所有状态只保存在本地，不产生隐私收集。
5. 现有主页、项目路由、伪后台和部署路径无回归。
6. 视觉保持黑、炭灰、灰白、暗金和少量暗红的原创异常档案身份。
